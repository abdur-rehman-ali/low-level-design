# Strategy Design Pattern
 
## What Is the Strategy Pattern?
 
The **Strategy Pattern** is a behavioral design pattern that defines a family of algorithms, encapsulates each one, and makes them interchangeable. It lets the algorithm vary independently from the clients that use it.
 
In other words: instead of hardcoding a behavior into a class, you extract it into separate "strategy" objects and inject whichever one you need at runtime.
 
---
 
## Core Concepts
 
- **Context** – the class that uses a strategy. It holds a reference to a strategy object and delegates the work to it.
- **Strategy (Interface/Abstract Class)** – defines the common interface all concrete strategies must implement.
- **Concrete Strategies** – the actual implementations of the algorithm.
---


## Example Setup.

```python
from abc import ABC, abstractmethod

# Behavioral/Strategy: Defines algorithm interface
class Strategy(ABC):
    @abstractmethod
    def execute(self, data):
        pass

# Concrete strategies - different behaviors
class ConcreteStrategyA(Strategy):
    def execute(self, data):
        return f"Strategy A processing {data}"

class ConcreteStrategyB(Strategy):
    def execute(self, data):
        return f"Strategy B processing {data}"

# Context - uses strategy
class Context:
    def __init__(self, strategy: Strategy):
        self._strategy = strategy
    
    def set_strategy(self, strategy: Strategy):  # Change behavior at runtime
        self._strategy = strategy
    
    def do_something(self, data):  # Delegates to strategy
        return self._strategy.execute(data)

# Usage - behavior can change at runtime
context = Context(ConcreteStrategyA())
print(context.do_something("data"))  # Strategy A processing data

context.set_strategy(ConcreteStrategyB())  # Swapping behavior
print(context.do_something("data"))  # Strategy B processing data

```

## Pros and Cons
 
### ✅ Pros
 
| Advantage | Description |
|---|---|
| **Open/Closed Principle** | Add new strategies without modifying the context class. |
| **Eliminates conditionals** | Replaces large `if/elif` chains with clean, interchangeable objects. |
| **Runtime flexibility** | Switch algorithms on the fly without changing the client. |
| **Testability** | Each strategy can be unit-tested in isolation. |
| **Single Responsibility** | Each class or function has one job — one algorithm. |
| **Reusability** | Strategies can be reused across different contexts. |
 
### ❌ Cons
 
| Disadvantage | Description |
|---|---|
| **Increased number of classes** | Each strategy adds a new class, which can bloat the codebase. |
| **Client awareness** | The client must know which strategies exist to choose one. |
| **Overkill for simple cases** | If there are only 2–3 algorithms and they rarely change, a simple `if/else` is cleaner. |
| **Communication overhead** | Context and strategies may need to share state, increasing coupling. |
 
---

## When to Use the Strategy Pattern
 
### ✔ Good Use Cases
 
- **Multiple payment methods** – credit card, PayPal, crypto, bank transfer.
- **Sorting/searching algorithms** – swap algorithms based on data size or type.
- **Data compression** – ZIP, GZIP, BZIP2 selected at runtime.
- **Authentication strategies** – JWT, OAuth, API Key, session-based.
- **Discount/pricing rules** – seasonal, membership, coupon-based pricing.
- **Routing strategies** – fastest route, shortest route, least traffic.
- **Validation logic** – different validation rules for different form types.
- **Export formats** – CSV, JSON, XML, PDF generated by different strategies.
### ✘ Avoid When
 
- You only have one algorithm and it won't change.
- The number of strategies is very small and stable.
- The algorithms are tightly coupled to the context's data (consider Template Method instead).
---

 
## Comparison with Related Patterns
 
| Pattern | Intent |
|---|---|
| **Strategy** | Interchangeable algorithms; chosen at runtime. |
| **Template Method** | Fixed algorithm skeleton; subclasses fill in the steps. |
| **State** | Object changes behavior based on internal state; strategies usually don't track state. |
| **Command** | Encapsulates a request/action; Strategy encapsulates an algorithm. |
| **Decorator** | Adds behavior on top; Strategy replaces behavior entirely. |
 
---
 
## Summary
 
The Strategy pattern is one of the most practical patterns in everyday software development. It shines whenever you have **multiple ways to do something** and want to keep your code clean, extensible, and easy to test. Python makes it especially ergonomic — you can use full class hierarchies for complex cases or simple callables for lightweight scenarios.
 
> **Rule of thumb:** If you're writing a long `if/elif` to choose between algorithms or behaviors, the Strategy pattern is probably the right fix.
